# PDF-Sign Package Rebranding Complete

## Issue

The `@signtusk/pdf-sign` package was still using the old Documenso naming internally:

- Cargo package name: `documenso_pdf-sign`
- README references: `@documenso/pdf-sign`
- Build output: "Compiling documenso_pdf-sign v0.0.0"

## Changes Made

### 1. Cargo.toml

```toml
[package]
edition = "2021"
name = "signtusk_pdf-sign"    # Changed from documenso_pdf-sign
version = "0.1.0"              # Updated from 0.0.0
```

### 2. Cargo.lock

```toml
[[package]]
name = "signtusk_pdf-sign"     # Changed from documenso_pdf-sign
version = "0.1.0"              # Updated from 0.0.0
```

### 3. README.md

Updated all references:

- Package name: `@signtusk/pdf-sign`
- Import statements: `require('@signtusk/pdf-sign')`
- Description: "SignTusk platform" instead of "Documenso platform"

## Build Output Now Shows

```
@signtusk/pdf-sign:build: Compiling signtusk_pdf-sign v0.1.0 (/app/packages/pdf-sign)
```

Instead of:

```
@signtusk/pdf-sign:build: Compiling documenso_pdf-sign v0.0.0 (/app/packages/pdf-sign)
```

## Files Updated

1. `packages/pdf-sign/Cargo.toml` - Package name and version
2. `packages/pdf-sign/Cargo.lock` - Package reference
3. `packages/pdf-sign/README.md` - Documentation and examples

## Why This Matters

1. **Consistency**: Package name now matches across npm and Rust
2. **Branding**: All references now use SignTusk instead of Documenso
3. **Version**: Updated to 0.1.0 to reflect it's a working release
4. **Build logs**: Clearer what's being compiled

## Next Build

When you rebuild, you'll see:

```
✅ Compiling signtusk_pdf-sign v0.1.0
✅ Finished release [optimized] target(s)
```

## Deploy

These changes are cosmetic but important for branding. Include them in your next deployment:

```bash
git add packages/pdf-sign/Cargo.toml packages/pdf-sign/Cargo.lock packages/pdf-sign/README.md
git commit -m "rebrand: update pdf-sign package from documenso to signtusk"
git push origin main
```

## Note on npm Subpackages

The `packages/pdf-sign/npm/*/README.md` files still reference `@documenso/pdf-sign`. These are auto-generated by the NAPI build system and will be updated automatically when you run:

```bash
cd packages/pdf-sign
npm run build
```

These don't affect functionality - they're just documentation for the platform-specific binary packages.
